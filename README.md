# 🏦 PostgreSQL Bank Cashback System 💳

[![PostgreSQL](https://img.shields.io/badge/PostgreSQL-316192?style=for-the-badge&logo=postgresql&logoColor=white)](https://www.postgresql.org/)
[![SQL](https://img.shields.io/badge/SQL-003B57?style=for-the-badge&logo=sqlite&logoColor=white)](#)

Реализация бэкенд-логики (на уровне СУБД) для системы расчета и начисления кэшбэка по карточным операциям клиентов банка. Проект представляет собой доработку существующей бэк-офисной базы данных банка (`dbBank`).

## 📋 Описание бизнес-логики

Процесс начисления кэшбэка в системе разделен на два этапа:
1. **Плановое начисление:** в момент совершения покупки (вставка в `CardOperations`), система определяет категорию по MCC-коду продавца, рассчитывает сумму вознаграждения и формирует *плановую* проводку (статус `is_plan = true`).
2. **Фактическая выплата:** ежедневно ночью запускается автоматический процесс (Хранимая процедура), который переводит плановые проводки в фактические (`is_plan = false`) для тех клиентов, у которых наступила дата расчета по договору обслуживания.

Также реализован API-уровень в виде табличных функций (UDF) для выдачи детализированных и сводных отчетов в мобильное приложение/веб-кабинет клиента.

## 🛠 Технологический стек и навыки

В проекте продемонстрировано уверенное владение продвинутыми возможностями PostgreSQL:
- **Основы:** DDL (создание таблиц, PK/FK, Constraints), DML.
- **Бизнес-логика в БД:** Триггеры (`AFTER INSERT`), Хранимые процедуры (`PROCEDURE`), Табличные функции (`RETURNS TABLE`).
- **Продвинутый SQL:** 
  - Оконные функции (`SUM() OVER`, `LAG() OVER`) для расчета нарастающего итога и процентов прироста.
  - Обобщенные табличные выражения (`CTE`).
  - Сложные объединения (`LEFT JOIN`, `JOIN`) и агрегации.
  - Динамическое вычисление без хардкода (поиск ID через справочники).

## 📂 Структура репозитория

Скрипты пронумерованы в порядке их логического выполнения (накатывания миграций):

| Скрипт | Описание |
| :--- | :--- |
| `01_create_cashback_tables.sql` | **DDL:** создание таблиц справочника категорий (`cashback_categories`) и маппинга MCC-кодов (`cashback_categories_mcc`). |
| `02_insert_cashback_data.sql` | **DML:** наполнение справочников первичными данными ("Авиабилеты", "Аптеки", "Продукты") и связывание их с MCC. |
| `03_create_cashback_trigger.sql` | **Triggers:** создание триггерной функции `create_cashback_tran_plan()`, которая перехватывает новые операции и автоматически генерирует плановые проводки. |
| `04_create_process_procedure.sql` | **Procedures:** хранимая процедура `process_cashback(calc_date)`, предназначенная для вызова планировщиком (cron) для начисления фактического кэшбэка. |
| `05_func_detailed_report.sql` | **Functions / Window Functions:** табличная функция `get_cashback_detailed()` для генерации подробного отчета с нарастающим итогом. |
| `06_func_summary_report.sql` | **Functions / CTE:** табличная функция `get_cashback_agg()` для генерации сводного отчета по категориям (с обработкой `NULL` и динамической фильтрацией). |

## 🚀 Ключевые особенности реализации

- **Отсутствие хардкода:** в триггерах и процедурах исключено жесткое прописывание ID. Все идентификаторы банков, валют (`RUB`) и аналитических признаков определяются динамически через `SELECT` к справочникам.
- **Безопасность данных:** реализованы проверки на `NULL` перед вставкой записей, а также установлены `CHECK` ограничения на уровне таблиц (например, процент кэшбэка от 0 до 100).
- **Оптимизация аналитики:** использование математических вычислений внутри оконных функций позволяет получать процент прироста относительно предыдущей операции за один проход таблицы, избегая тяжелых `Self Join` и подзапросов.
